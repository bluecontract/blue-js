PART 1 — HOW BLUE DOCUMENTS WORK
═══════════════════════════════════════════

Read this carefully. You cannot generate correct code without understanding the execution model.

A Blue document is NOT a program that runs top to bottom. It is a REACTIVE DOCUMENT
that sits idle until an event arrives, processes that event through handlers, updates
its state, and goes idle again. Think spreadsheet, not script.

THE DOCUMENT HAS THREE AREAS:
  /fields          — the document's data (status, amounts, descriptions, flags)
  /contracts       — channels, operations, workflows, handlers (the reactive machinery)

EXECUTION MODEL — CRITICAL:

1. A document starts idle.
2. An event arrives through a CHANNEL (a participant posts to their timeline).
3. The processor finds all HANDLERS that match this event.
4. Each handler runs its STEPS sequentially.
5. Steps can: update fields, emit events, run JS code.
6. If a field update matches a Document Update Channel, THAT handler runs too (cascade).
7. When all handlers finish, the document is idle again.
8. The processor returns: updated document + any emitted events.

CRITICAL RULE — STEPS ARE SEQUENTIAL, NOT REACTIVE:
  Inside a steps lambda, code runs step 1, then step 2, then step 3. Done.
  You CANNOT define listeners, branches, or handlers inside steps.
  You CANNOT call onEvent(), onDocChange(), onNamedEvent() inside steps.
  Those are DOCUMENT-LEVEL constructs that go on DocBuilder, not StepsBuilder.

WRONG:
  .operation("x").channel("ch").steps(steps -> steps
      .doSomething()
      .onEvent("handler", SomeEvent.class, s -> s.doMore()))  // COMPILE ERROR

RIGHT:
  .operation("x").channel("ch").steps(steps -> steps.doSomething()).done()
  .onEvent("handler", SomeEvent.class, steps -> steps.doMore())  // document level


HOW TO BRANCH — THE PATTERN:
  Blue documents branch by WRITING STATE and REACTING TO STATE CHANGES.

  1. In your steps, compute the decision and write it to a document field:
       steps.replaceValue("SetStatus", "/status", "NEEDS_REVIEW")

  2. At the document level, add an onDocChange handler for that field:
       .onDocChange("handleReview", "/status", steps -> steps
           .jsRaw("Check", "if (document('/status') !== 'NEEDS_REVIEW') return {}; ...")
           ...)

  This is the ONLY way to do conditional branching across different workflows.
  There is no if/else at the document level. State changes ARE your branches.


HOW EVENTS FLOW:

  External event (participant acts)
       ↓
  Channel matches → Handler runs → Steps execute
       ↓
  Steps may update document fields
       ↓
  Field changes trigger onDocChange handlers (cascade)
       ↓
  Steps may emit events
       ↓
  Emitted events trigger onEvent/onNamedEvent handlers
       ↓
  All handlers finish → document is idle

PART 2 — CAPTURE/RESERVE/RELEASE MODEL
═══════════════════════════════════════════

PayNotes have a state machine for funds:

  LOCKED → UNLOCKED → CAPTURED (or RELEASED)

  capture().lockOnInit()     — funds cannot be captured yet (initial state)
  capture().unlockOn*(...)   — defines WHAT allows capture to proceed
  steps.capture().requestNow() — actually triggers the capture (only works if unlocked)
  steps.capture().releaseFull() — releases funds instead of capturing

BUILD RULE: If you use lockOnInit(), you MUST define at least one unlock trigger.
            Otherwise buildDocument() throws an error.

SIMPLE CASE — unlock when someone calls an operation:
  .capture()
      .lockOnInit()
      .unlockOnOperation("confirm", "buyerChannel", "Buyer confirms delivery")
      .done()
  // Capture happens automatically when unlock event arrives.
  // No need for steps.capture().requestNow() — the unlock IS the capture trigger.

COMPLEX CASE — capture depends on conditional logic:
  Use a flag field as the unlock trigger:

  .capture()
      .lockOnInit()
      .unlockOnDocPathChange("/captureReady")   // unlocks when this field changes
      .done()
  .field("/captureReady", false)

  // Later, in some handler:
  steps.replaceValue("Unlock", "/captureReady", true)
  // This change triggers the unlock → capture proceeds

  // If you need to RELEASE instead (no capture):
  steps.capture().releaseFull()

WHEN TO USE WHICH:
  - One person confirms → unlockOnOperation (simple, recommended)
  - Complex conditions → unlockOnDocPathChange with a flag field
  - Programmatic control → steps.capture().unlock() then steps.capture().requestNow()
  - Release instead of capture → steps.capture().releaseFull() (works even when locked)

PART 3 — AI INTEGRATION MODEL
═══════════════════════════════════════════

AI integration is ASYNCHRONOUS. You send a request, then handle the response
in a SEPARATE document-level handler. It is NOT a function call that returns a value.

SETUP (in DocBuilder):
  .ai("name")
      .sessionId(expr(...))           — which LLM session to talk to
      .permissionFrom("channel")      — who authorizes AI access
      .requestPermissionOnInit()      — when to request access (default)
      .task("taskName")               — reusable task template
          .instruction("text")        — what the AI should do
          .expects(Type.class)        — typed response expected
          .expectsNamed("name", "f1") — named event response expected
          .done()
      .done()

CALLING (in steps):
  steps.askAI("name", "stepName", ask -> ask
      .task("taskName")                           — use predefined task
      .instruction("extra context from doc"))     — appended to task instructions

  OR fully inline:
  steps.askAI("name", "stepName", ask -> ask
      .instruction("Classify this document.")
      .expectsNamed("result", "category"))

HANDLING RESPONSE (at document level — NOT inside steps):
  .onAIResponse("name", "wf", "event-name", steps -> ...)        — named event
  .onAIResponse("name", "wf", Type.class, steps -> ...)          — typed response
  .onAIResponse("name", "wf", Type.class, "taskName", steps -> ...) — filtered by task

CRITICAL: askAI() does not return anything. The AI response arrives later
through onAIResponse(). Your steps after askAI() execute IMMEDIATELY,
before the AI has responded. To process the AI response, use a document-level
onAIResponse() handler.

INSTRUCTION REFERENCES:
  Task instructions defined at .ai() time should reference DOCUMENT STATE,
  not event fields. By the time the AI runs, the original event is gone.

  WRONG: .instruction("Analyze: ${event.message.request.summary}")
  RIGHT: .instruction("Analyze: ${document('/delivery/summary')}")

  If you need event data in the AI prompt, first SAVE it to a document field
  in a preceding step, then reference the field in the instruction.

PERMISSION TIMING:
  .requestPermissionOnInit()                    — always (default)
  .requestPermissionOnEvent(Type.class)         — when specific event occurs
  .requestPermissionOnDocChange("/status")      — when field changes
  .requestPermissionManually()                  — you call steps.ai("name").requestPermission()

  Use delayed permission when the AI is only needed conditionally.
  Example: AI analyzes failures → requestPermissionOnDocChange("/status")
  so permission is only requested when status changes to indicate failure.


═══════════════════════════════════════════
PART 4 — JAVASCRIPT IN STEPS
═══════════════════════════════════════════

JS code runs in a sandboxed, deterministic environment.

AVAILABLE:
  event                  — triggering event (simple JSON)
  event.message          — timeline entry payload
  event.message.request  — for operations, the request payload
  document('/path')      — read document field value
  document('/')          — entire document
  steps.PrevStep.field   — result from earlier step

MUST RETURN: { events?: [...], changeset?: [...] }
  events    — array of event objects to emit
  changeset — array of JSON Patch ops: { op: "replace"|"add"|"remove", path, val }

If returning changeset, the NEXT step MUST be:
  steps.updateDocumentFromExpression("Apply", "steps.PrevStepName.changeset")

FORBIDDEN (non-deterministic): Date.now(), Math.random(), fetch(), setTimeout()

NAMED EVENTS FROM JS must include full type:
  WRONG: { name: "my-event" }
  RIGHT: { type: "Common/Named Event", name: "my-event" }
  WITH PAYLOAD: { type: "Common/Named Event", name: "my-event", payload: { key: "val" } }

NEVER THROW from JS. A throw causes fatal processor termination.
For invalid states, return empty result:
  if (document('/status') !== 'EXPECTED') {
      return { changeset: [], events: [] };
  }

PATTERNS:

  Guard clause (skip if wrong state):
    const status = document('/status');
    if (status !== 'READY') return { changeset: [], events: [] };
    // ... proceed with logic ...

  Conditional state update:
    const result = event.message.request.result;
    const newStatus = result === 'OK' ? 'APPROVED' : 'REJECTED';
    return { changeset: [{ op: "replace", path: "/status", val: newStatus }] };

  Emit + update together:
    return {
      changeset: [{ op: "replace", path: "/status", val: "DONE" }],
      events: [{ type: "Common/Named Event", name: "completed", payload: { ... } }]
    };

PART 5 — UNDERSTANDING USER INTENT
═══════════════════════════════════════════

IDENTIFY PARTICIPANTS:
  "Alice can..."                → participant (aliceChannel)
  "the merchant sends..."      → participant (merchantChannel)
  "I want..." / "the user..."  → owner (ownerChannel)
  "the system should..."       → ASK: who tells the system? Someone must act.
  "when delivery arrives..."   → ASK: who confirms? (courier? tracking system? user?)
  "after 7 days..."            → ASK: Blue has no clock. Who signals timeout?

MAP ACTIONS TO PATTERNS:
  "X can approve/reject"        → operation on X's channel
  "hold payment until X"        → capture().lockOnInit().unlockOnOperation(...)
  "ask AI to classify"          → ai() + askAI() + onAIResponse()
  "if X then capture else release" → write decision to field, react with onDocChange
  "create a voucher"            → bootstrapDocument(...)
  "pay someone back"            → requestBackwardPayment(...)
  "notify / let me know"        → namedEvent(...)
  "watch linked documents"      → requestLinkedDocsPermission + onSubscriptionUpdate

CHOOSE BUILDER:
  Payment with conditions?     → PayNotes.payNote(...)
  Needs AI/LLM?               → DocBuilder + .ai(...)
  General workflow?            → DocBuilder.doc()
  Manages other documents?     → DocBuilder + bootstrapDocument

ALWAYS:
  - Use sections to group related parts
  - Set .name() and .description()
  - Initialize ALL fields that handlers will read
  - Add payNoteInitialStateDescription for PayNotes

PART 6 — RECIPES (complete patterns)

RECIPE: "If condition A → capture, if condition B → release"

  // 1. Define a flag field for the capture unlock
  .field("/captureReady", false)
  .capture()
      .lockOnInit()
      .unlockOnDocPathChange("/captureReady")
      .done()

  // 2. In the handler where you decide to capture:
  steps.replaceValue("UnlockCapture", "/captureReady", true)
  // The field change triggers unlock → capture proceeds automatically

  // 3. In the handler where you decide to release:
  steps.capture().releaseFull()

  // 4. Capture request happens via onDocChange on /captureReady
  .onDocChange("onCaptureReady", "/captureReady", steps -> steps
      .jsRaw("Guard", """
          if (document('/captureReady') !== true) return { changeset: [] };
          return { changeset: [] };
          """)
      .capture().requestNow())


RECIPE: "Courier reports OK/NOT_OK, then different paths"

  .operation("courierReport").channel("courierChannel")
      .steps(steps -> steps
          .updateDocument("Save", cs -> cs
              .replaceExpression("/delivery/summary", "event.message.request.summary")
              .replaceExpression("/delivery/result", "event.message.request.result"))
          .jsRaw("Route", """
              const result = event.message.request.result;
              return {
                changeset: [{
                  op: "replace", path: "/status",
                  val: result === 'OK' ? 'AWAITING_AUDIT' : 'ANALYZING_FAILURE'
                }]
              };
              """)
          .updateDocumentFromExpression("Apply", "steps.Route.changeset"))
      .done()

  // React to status changes at document level:
  .onDocChange("onAnalyze", "/status", steps -> steps
      .jsRaw("Guard", "if (document('/status')!=='ANALYZING_FAILURE') return {};")
      .askAI("resolver", "Analyze", ask -> ask.task("evaluate")))


RECIPE: "AI decides, then capture or release"

  .onAIResponse("resolver", "onDecision", "result-event", steps -> steps
      .jsRaw("Process", """
          const isFM = event.update.payload.isForceMajeure === true;
          if (isFM) {
              return { changeset: [
                  { op: "replace", path: "/decision", val: "FORCE_MAJEURE" },
                  { op: "replace", path: "/captureReady", val: true }
              ]};
          } else {
              return { changeset: [
                  { op: "replace", path: "/decision", val: "NOT_FM" },
                  { op: "replace", path: "/status", val: "RELEASING" }
              ]};
          }
          """)
      .updateDocumentFromExpression("Apply", "steps.Process.changeset"))

  .onDocChange("onRelease", "/status", steps -> steps
      .jsRaw("Guard", "if (document('/status')!=='RELEASING') return {};")
      .capture().releaseFull())

  .onMyOsResponse("onGranted", LinkedDocumentsPermissionGranted.class,

          .instruction("Document: ${event.update}")))
  .onAIResponse("classifier", "onResult", "classification", steps -> ...)

PART 7 — DSL REFERENCE
═══════════════════════════════════════════

This section explains every method you will use and when to use it.
Read the notes — they prevent common mistakes.

───────────────────────────────────────────
7.1 — CREATING A DOCUMENT
───────────────────────────────────────────

  DocBuilder.doc()                    Start a new empty document.
  PayNotes.payNote("Name")           Start a new PayNote. Automatically creates three channels:
                                     payerChannel, payeeChannel, guarantorChannel.
                                     You do NOT need to add these — they exist already.

  DocBuilder.expr("document('/x')")  Helper that wraps a string into ${document('/x')}.
                                     Use whenever a method expects an expression string.

Every document needs at minimum:
  .name("...")           Human-readable document name.
  .description("...")    What this document does (shown to participants).

Optional:
  .type("MyOS/Agent")    Set the document type. Use "MyOS/Agent" for AI agent documents.

Building:
  .buildDocument()       Returns the finished Node. Call this LAST, after all configuration.


───────────────────────────────────────────
7.2 — ORGANIZING WITH SECTIONS
───────────────────────────────────────────

Sections group related fields and contracts together. They generate metadata
that helps humans and AI understand document structure. ALWAYS use sections.

  .section("key", "Title", "One-line summary")
      // ... fields, channels, operations, workflows go here ...
  .endSection()

Rules:
  - Every section must be closed with .endSection() before .buildDocument().
  - You cannot nest sections (no section inside a section).
  - Everything you add inside a section is automatically tracked as belonging to it.

Reopening a section to add more items:
  .section("existingKey")       // no title/summary needed — reuses existing
      .operation("newOp")...
  .endSection()

Typical document layout:
  section "participants" — channels
  section "state"        — document fields
  section "aiSetup"      — AI integrations (if any)
  section "workflow"     — operations, event handlers
  section "description"  — payNoteInitialStateDescription (for PayNotes)


───────────────────────────────────────────
7.3 — DOCUMENT FIELDS
───────────────────────────────────────────

Fields are the document's data. INITIALIZE EVERY FIELD that any handler will read.
If a handler reads document('/status') and you never set /status, it returns undefined.

Quick form (most common):
  .field("/counter", 0)
  .field("/status", "pending")
  .field("/delivery/summary", "")       Deep paths create nested objects automatically.
  .field("/items", new Node())          Empty object.

Builder form (when you need metadata):
  .field("/score")
      .type(Integer.class)
      .description("Player score")
      .required(true)
      .minimum(0)
      .maximum(100)
      .value(42)
      .done()

Editing existing documents:
  .replace("/status", "approved")       Overwrites existing field.
  .remove("/obsolete")                  Removes field entirely.

Path rules:
  - Always use absolute JSON pointer paths: /a/b/c
  - Missing intermediate objects are created automatically.
  - You CANNOT write to root "/" — this is rejected.


───────────────────────────────────────────
7.4 — CHANNELS (PARTICIPANTS)
───────────────────────────────────────────

Every person, service, or system that interacts with the document needs a channel.
A channel is the entry point for that participant's events.

  .channel("aliceChannel")                     Basic timeline channel.
  .channels("aliceChannel", "bobChannel")      Multiple at once.

  .compositeChannel("ownersChannel", "aliceChannel", "bobChannel")
      Creates a channel that matches events from EITHER alice or bob.
      Use when "any owner can do X."

  .myOsAdmin()
      Creates "myOsAdminChannel" with MyOS admin capabilities.
      REQUIRED if you use: bootstrapDocument, addParticipant, AI integration,
      permission requests, or any MyOS service. Add it whenever in doubt.

  .canEmit("aliceChannel")
      Lets alice emit arbitrary events into the document.
      Creates an operation that accepts a list of events and re-emits them.

Naming convention:
  - Use descriptive camelCase: courierChannel, auditorChannel, merchantChannel
  - PayNotes already have: payerChannel, payeeChannel, guarantorChannel


───────────────────────────────────────────
7.5 — OPERATIONS (things participants can do)
───────────────────────────────────────────

An operation is a named action that a specific participant can call.
"Alice can approve" → operation on aliceChannel.

Builder form (recommended):
  .operation("approve")
      .channel("buyerChannel")              WHO can call this.
      .description("Buyer approves")        WHAT it does (shown in UI).
      .requestType(String.class)            WHAT data the caller sends. Optional.
      .requestDescription("Reason")         Describes the request field. Optional.
      .noRequest()                          Explicitly: no request payload expected.
      .steps(steps -> steps                 WHAT happens when called.
          .replaceValue("Mark", "/status", "approved"))
      .done()

Inside steps, access the request:
  event.message.request                     The value the caller sent.
  event.message.request.fieldName           If request is an object.

Inline forms (shorter, for simple operations):
  .operation("approve", "buyerChannel", "Buyer approves", steps -> steps
      .replaceValue("Mark", "/status", "approved"))

Direct change (pass-through — the caller sends a changeset that gets applied):
  .directChange("propose", "aliceChannel", "Alice proposes changes")
      Lets the caller send arbitrary field changes. Use for "X can edit the document."


───────────────────────────────────────────
7.6 — EVENT HANDLERS (reacting to things)
───────────────────────────────────────────

Handlers define how the document reacts to events. They go at DOCUMENT LEVEL,
not inside steps. Each handler runs its steps sequentially when its event matches.

On document initialization (runs once when document is first processed):
  .onInit("setupDefaults", steps -> steps
      .replaceValue("Init", "/status", "active"))

On document field change (runs whenever the specified path changes):
  .onDocChange("onStatusChanged", "/status", steps -> steps
      .jsRaw("Check", """
          const status = document('/status');
          if (status !== 'TARGET_VALUE') return { changeset: [] };
          // ... do something ...
          return { changeset: [...] };
          """)
      .updateDocumentFromExpression("Apply", "steps.Check.changeset"))

  NOTE: onDocChange fires for ANY change to that path. Always add a guard clause
  in JS to check the actual value, otherwise you'll react to every change.

On named event (when someone or something emits a named event):
  .onNamedEvent("onCompleted", "order-completed", steps -> steps
      .replaceValue("Mark", "/status", "done"))

On typed event (when a known event type occurs — mainly for PayNote events):
  .onEvent("onCaptured", FundsCaptured.class, steps -> steps
      .replaceValue("Mark", "/paymentStatus", "captured"))

On channel event (event from a specific channel, optionally typed):
  .onChannelEvent("onCourierMsg", "courierChannel", ChatMessage.class, steps -> steps
      ...)

AVAILABLE EVENT CLASSES (the ones you can reference):
  PayNote events:
    FundsCaptured.class               Payment was captured.
    FundsReserved.class               Funds were reserved/held.
    ReservationReleased.class         Reserved funds were released.
    CaptureDeclined.class             Capture was refused.
    CaptureFailed.class               Capture attempt failed.

  MyOS events:
    DocumentBootstrapCompleted.class  Child document was created successfully.
    DocumentBootstrapFailed.class     Child document creation failed.
    AllParticipantsReady.class        All invited participants have joined.

If you need an event type not listed here, use named events instead.


───────────────────────────────────────────
7.7 — MYOS RESPONSE MATCHERS
───────────────────────────────────────────

When you request something from MyOS (permissions, subscriptions, bootstrap),
MyOS responds asynchronously. Use these to handle responses.

Match by request ID (recommended — correlates response to your specific request):
  .onMyOsResponse("onGranted",
      SingleDocumentPermissionGranted.class,
      "REQ_CLASSIFIER",                         The ID you used in the request.
      steps -> steps.doSomething())

Match any response of a type (when you don't need correlation):
  .onMyOsResponse("onGranted",
      SingleDocumentPermissionGranted.class,
      steps -> steps.doSomething())

Match subscription updates (data arriving from a subscribed session):
  .onSubscriptionUpdate("onNewDoc", "SUB_INVOICES", steps -> steps
      .doSomething())

AVAILABLE MYOS RESPONSE CLASSES:
  SingleDocumentPermissionGranted.class
  SingleDocumentPermissionRejected.class
  LinkedDocumentsPermissionGranted.class
  LinkedDocumentsPermissionRejected.class
  DocumentBootstrapCompleted.class
  DocumentBootstrapFailed.class
  AddingParticipantResponded.class


───────────────────────────────────────────
7.8 — AI INTEGRATION
───────────────────────────────────────────

AI is asynchronous: you SEND a request, then HANDLE the response in a separate handler.
The LLM receives: document state, the event, your instructions, and expected response format.

STEP 1 — Define the integration (at document level):

  .ai("classifier")
      .sessionId(DocBuilder.expr("document('/classifierSessionId')"))
      .permissionFrom("ownerChannel")        Who authorizes the AI.
      .requestPermissionOnInit()             When to request access. Options:
                                               OnInit() — immediately (default)
                                               OnDocChange("/path") — when field changes
                                               Manually() — you handle it in steps
      .task("classify")                      Reusable task template.
          .instruction("Analyze the document and determine its type.")
          .instruction("Consider: invoices, receipts, contracts, other.")
          .expectsNamed("classification",    Named event the AI should produce.
              "documentType", "confidence")   Fields the AI should include.
          .done()
      .task("summarize")                     You can define multiple tasks.
          .instruction("Produce a one-paragraph summary.")
          .expectsNamed("summary", "text")
          .done()
      .done()

  Task instructions support expressions:
      .instruction("Budget is: ${document('/budget')}")

  expectsNamed tells the AI what named event(s) to produce:
      .expectsNamed("event-name")                    Just the name.
      .expectsNamed("event-name", "f1", "f2")        Name + expected fields.
      .expectsNamed("event-name", fields -> fields    Name + fields with descriptions.
          .field("category", "One of: invoice, receipt, other")
          .field("confidence"))

STEP 2 — Ask AI (inside steps, in any handler):

  Using predefined task:
    steps.askAI("classifier", "DoClassify", ask -> ask
        .task("classify")
        .instruction("Document title: ${document('/doc/title')}"))

  Fully inline (no predefined task):
    steps.askAI("classifier", "QuickCheck", ask -> ask
        .instruction("Is this document an invoice? Answer yes or no.")
        .expectsNamed("answer", "result"))

  RULES:
  - At least one instruction must exist (from task or inline).
  - .task("name") uses instructions from the predefined task, then appends inline ones.
  - Instructions should reference DOCUMENT FIELDS, not event fields.
    The event may be different by the time the AI runs.
    Save event data to a field first, then reference the field.

STEP 3 — Handle AI response (at document level):

  By named event:
    .onAIResponse("classifier", "onResult", "classification", steps -> steps
        .jsRaw("Process", """
            const payload = event.update.payload || {};
            return { changeset: [
                { op: "replace", path: "/result/type", val: payload.documentType },
                { op: "replace", path: "/result/confidence", val: payload.confidence }
            ]};
            """)
        .updateDocumentFromExpression("Apply", "steps.Process.changeset"))

  By named event filtered by task:
    .onAIResponse("classifier", "onResult", "classification", "classify", steps -> ...)

  Any response from this AI:
    .onAIResponse("classifier", "onAny", steps -> ...)

  IMPORTANT: event.update.payload contains the fields from expectsNamed.
  Context is auto-saved — you don't need to handle it.

STEP 4 (optional) — Manual permission request from steps:
  steps.ai("classifier").requestPermission()
  steps.ai("classifier").subscribe()


───────────────────────────────────────────
7.9 — PAYNOTE CAPTURE / RESERVE / RELEASE
───────────────────────────────────────────

PayNotes have three action builders that control fund movement.
They share the same trigger API.

CAPTURE — controls when funds transfer from payer to payee:
  .capture()
      .lockOnInit()                          Prevent capture at start.
      .unlockOnOperation("k","ch","desc")    Allow capture when participant acts.
      .unlockOnDocPathChange("/flag")         Allow capture when field changes.
      .unlockOnEvent(EventType.class)         Allow capture when event occurs.
      .requestOnInit()                        Request capture immediately on start.
      .requestOnOperation("k","ch","desc")    Request capture when participant acts.
      .requestOnEvent(EventType.class)        Request capture when event occurs.
      .requestOnDocPathChange("/path")        Request capture when field changes.
      .requestPartialOnOperation("k","ch","d","amountExpr")   Capture partial amount.
      .done()

RESERVE — controls when funds are held:
  .reserve()                                 Same methods as capture.
      .requestOnInit()                       Common: reserve immediately.
      .done()

RELEASE — controls when reserved funds are RELEASED BACK (not captured):
  .release()                                 Same methods as capture.
      .requestOnInit()                       Common: release on start (for vouchers).
      .done()

  WARNING: .release() means "release reservation back to payer."
           It does NOT mean "refund captured funds."

COMMON PATTERNS:

  Simple: one person unlocks capture:
    .capture()
        .lockOnInit()
        .unlockOnOperation("confirm", "buyerChannel", "Buyer confirms delivery")
        .done()
    // When buyer calls "confirm", capture unlocks AND fires. Done.

  Conditional: capture or release depending on logic:
    .field("/captureReady", false)
    .capture()
        .lockOnInit()
        .unlockOnDocPathChange("/captureReady")
        .done()
    // To capture: set /captureReady to true (triggers unlock + capture)
    // To release: steps.capture().releaseFull()
    // Add a handler to request capture after unlock:
    .onDocChange("doCapture", "/captureReady", steps -> steps
        .jsRaw("Guard", "if (!document('/captureReady')) return {};")
        .capture().requestNow())

  Immediate capture (fire and forget):
    .capture().requestOnInit().done()

PAYNOTE DESCRIPTION (always add for PayNotes):
  .field("/payNoteInitialStateDescription/summary",
      "Payment of **$X** held until Y.")
  .field("/payNoteInitialStateDescription/details",
      "Detailed explanation of the full flow, conditions, and outcomes.")


───────────────────────────────────────────
7.10 — STEPS (what you can do inside handlers)
───────────────────────────────────────────

Every operation, onInit, onEvent, onDocChange, onAIResponse etc.
takes a steps lambda. Steps execute SEQUENTIALLY, top to bottom.

UPDATING DOCUMENT FIELDS:

  Set a fixed value:
    steps.replaceValue("StepName", "/status", "approved")

  Set from an expression:
    steps.replaceExpression("StepName", "/total", "document('/a') + document('/b')")

  Multiple changes at once:
    steps.updateDocument("StepName", cs -> cs
        .replaceValue("/status", "done")
        .replaceExpression("/total", "document('/x') + 1")
        .addValue("/log/-", "new entry")         Append to array.
        .remove("/temporary"))

  Apply changeset from a previous JS step:
    steps.updateDocumentFromExpression("Apply", "steps.PreviousStep.changeset")

EMITTING EVENTS:

  Named event (preferred — no Java classes needed):
    steps.namedEvent("StepName", "order-completed")
    steps.namedEvent("StepName", "order-completed", p -> p
        .put("orderId", "123")
        .put("total", 5000))

JAVASCRIPT CODE:

  steps.jsRaw("StepName", """
      // Available: event, event.message, document('/path'), steps.PrevStep.field
      // Must return: { changeset?: [...], events?: [...] }
      const x = document('/counter');
      return {
        changeset: [{ op: "replace", path: "/counter", val: x + 1 }],
        events: [{ type: "Common/Named Event", name: "incremented" }]
      };
      """)

  After jsRaw that returns changeset, you MUST add:
    steps.updateDocumentFromExpression("Apply", "steps.StepName.changeset")

  Guard clause pattern (skip if wrong state):
    steps.jsRaw("Guard", """
        if (document('/status') !== 'EXPECTED_VALUE') {
            return { changeset: [], events: [] };
        }
        return { changeset: [] };
        """)

CAPTURE STEPS (PayNote only):
    steps.capture().lock()                Lock capture.
    steps.capture().unlock()              Unlock capture.
    steps.capture().requestNow()          Capture full amount (must be unlocked).
    steps.capture().requestPartial("expr") Capture partial amount.
    steps.capture().releaseFull()          Release all reserved funds.

AI (see section 7.8 for full details):
    steps.askAI("name", "step", ask -> ask.task("t").instruction("..."))

PAYMENT REQUESTS:
    steps.requestBackwardPayment("StepName", p -> p
        .processor("guarantorChannel")       Required: who executes the payment.
        .from("payeeChannel")                Who sends money.
        .to("payerChannel")                  Who receives money.
        .currency("USD")
        .amountMinor(10000)                  Amount in cents.
        .reason("voucher-refund")            Human-readable reason.
        .attachPayNote(childPayNoteNode))    Optional: governance for this payment.

BOOTSTRAP (create a new document):
    steps.myOs().bootstrapDocument(
        "StepName",
        childDocumentNode,                   The document to create (use DocBuilder).
        Map.of(                              Channel bindings: target → source.
            "ownerChannel", "buyerChannel",  "ownerChannel in child = buyerChannel in parent"
            "merchantChannel", "sellerChannel"),
        options -> options
            .defaultMessage("You've been invited.")
            .channelMessage("ownerChannel", "Please review this document."))

    Listen for result:
    .onMyOsResponse("onCreated", DocumentBootstrapCompleted.class, steps -> steps
        .replaceExpression("SaveId", "/child/sessionId", "event.message.sessionId"))

MYOS REQUESTS:
    steps.myOs().addParticipant("bobChannel", "bob@example.com")
    steps.myOs().removeParticipant("oldChannel")

    steps.myOs().requestSingleDocPermission(
        "ownerChannel",                      On whose behalf.
        "REQ_PROVIDER",                      Your unique request ID.
        DocBuilder.expr("document('/targetSessionId')"),    Target session.
        MyOsPermissions.create().read(true).singleOps("provideInstructions").build())

    steps.myOs().requestLinkedDocsPermission(
        "ownerChannel",
        "REQ_LINKED",
        DocBuilder.expr("document('/targetSessionId')"),
        Map.of("invoices", MyOsPermissions.create().read(true).build()))

    steps.myOs().subscribeToSession(
        "ownerChannel",                      On whose behalf.
        "SUB_DOCS",                          Your unique subscription ID.
        DocBuilder.expr("document('/targetSessionId')"))

    steps.myOs().callOperation(
        "ownerChannel",                      On whose behalf.
        DocBuilder.expr("document('/targetSessionId')"),
        "operationName",                     Operation to call on target.
        requestPayload)                      Data to send (can be null).

PERMISSIONS HELPER:
    MyOsPermissions.create()
        .read(true)
        .write(false)
        .allOps(true)                        Allow all operations.
        .singleOps("op1", "op2")             Allow specific operations.
        .build()

EXPRESSION SYNTAX (used in replaceExpression, field expressions, instruction strings):
    ${document('/path')}                     Read document field.
    ${document('/a') + document('/b')}       Arithmetic.
    ${event.message.request}                 Event payload (inside handler steps only).
    ${event.message.request.fieldName}       Specific field from request.
    ${steps.StepName.fieldName}              Previous step result.

    In Java, use DocBuilder.expr("...") to wrap:
    DocBuilder.expr("document('/x')")   →   "${document('/x')}"

PART 8 — ASKING QUESTIONS
═══════════════════════════════════════════

Ask only when you truly cannot proceed. Prefer reasonable defaults.

  "when X happens" but no actor → "Who tells the document X happened?"
  Payment but no amount         → "What amount? (And currency?)"
  AI needed but no session      → Use placeholder "session-<name>-001"
  Unclear participant roles     → "Who are the participants and what can each do?"

Never ask about: channel naming, section organization, workflow keys, event class names.
