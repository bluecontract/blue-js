CORRECTION CYCLE 4 — PayNote operation-triggered branch schemas
====================

Goal:
Do not force artificial `Boolean` / `Integer` request schemas on operation-triggered PayNote macro branches if the runtime accepts omitted request schema for "any / empty" semantics.

User-provided direction:
If an operation should accept any Blue node, including empty, the operation request should normally be omitted rather than forced to `Boolean`/`Integer`.

Representative shape:
```yaml
sendPayNote:
  type: Conversation/Operation
  channel: merchantChannel
  description: Submit a PayNote Delivery for processing.

sendPayNoteImpl:
  type: Conversation/Sequential Workflow Operation
  operation: sendPayNote
  steps:
    - name: Request Delivery Bootstrap
      type: Conversation/JavaScript Code
      code: |
        return {
          events: [
            event.message.request
          ]
        };
```

Required changes:
1. Re-verify the current runtime semantics for operation-triggered macro branches.
2. For generic trigger branches (e.g. unlock/request on operation), prefer omitting `request` entirely when the semantics are "accept any/empty".
3. Re-check partial-amount branches:
   - if runtime proof shows they also work with omitted `request`, prefer omission
   - only keep explicit request schema if there is runtime-confirmed necessity
4. Remove the previous assumption that `Boolean` or `Integer` schemas are required by default.
5. Update Stage 6 tests/docs/deviations accordingly.

Acceptance:
- Operation-triggered PayNote branches no longer force artificial request schemas where omission is runtime-correct.
- Tests prove the chosen behavior.
- Docs match the actual runtime-confirmed semantics.
