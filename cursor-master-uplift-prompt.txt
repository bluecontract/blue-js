## Stage order and required outcomes

### Stage A — ergonomics and public export surfaces
Implement:
- a public alias-style JSON export:
  - `buildJson()` and/or `nodeToAliasJson(...)`
- `contract(key, contractLike)`
- optional `contracts(record)`

Requirements:
- external repos should no longer need local alias-json helpers
- external repos should no longer need `field('/contracts/...')` for normal contract insertion
- section tracking must work with the new contract helper(s)
- add docs and tests

### Stage B — generic authoring helpers and parity surface
Implement:
- `SimpleDocBuilder`
- `contractsPolicy(...)`
- `directChange(...)`
- `proposeChange(...)`
- `acceptChange(...)`
- `rejectChange(...)`
- `anchors(...)`
- `links(...)`
- `canEmit(...)`

Requirements:
- use Java as API/parity guide
- adapt to current runtime/mapping docs when parity conflicts with runtime
- document deviations narrowly and honestly
- add tests and docs

### Stage C — payment / conversation convenience APIs
Implement thin, runtime-confirmed convenience APIs such as:
- `triggerPayment(...)`
- `requestBackwardPayment(...)`
and any other clearly justified convenience APIs that reduce unnecessary raw fallbacks.

Requirements:
- keep them thin
- do not duplicate macro-builder behavior
- add tests and docs

### Stage D — macro-builder hardening with canonical scenarios
Harden real orchestration surfaces with stronger proof:
- `access(...)`
- `accessLinked(...)`
- `agency(...)`
- `ai(...).done()`
- `askAI(...)`
- `onAIResponse(...)`
- PayNote macro builders

Requirements:
- each major macro surface needs:
  - parity proof where available
  - runtime/integration proof
  - canonical scenario proof
- reduce raw fallback usage where the DSL can clearly express the behavior

### Stage E — architecture and maintainability uplift
Refactor internals for maintainability without breaking public API.
Focus on:
- splitting monolithic builders by domain
- clarifying shared internals
- preserving behavior
- keeping tests green

### Stage F — optional `myos-js` companion intake
If `references/alternative-sdk-serious/libs/myos-js/**` exists and is strong:
- evaluate it as a separate companion package
- import/adapt it only if quality and fit are high
- keep it separate from `sdk-dsl` runtime
- add docs/examples for interop

If it is absent or unsuitable:
- document deferment cleanly
- continue with the rest of the plan

### Stage G — final release hardening
Finalize:
- docs refresh
- examples refresh
- scorecard closure
- final QA sweep
- final repo-hygiene sweep

The whole plan is complete only when:
- every category in the scorecard is at 9 or above

- canonical scenarios remain green
- no major unresolved regression remains
- docs match the final implementation
